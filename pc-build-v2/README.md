# CR Tools V2

Isolated M7 release-capable client for the Windows-only CR Tools V2 application. This package does
not reuse V1 implementation code and has a separate development and packaging
identity:

- package: `cr-tools-v2`
- product: `CR Tools V2`
- application ID: `com.snipe.client.v2`
- artifact: `CR_Tools_V2_Setup_${version}.exe`

## Commands

Requires Node.js `^20.19.0` or `>=22.12.0` and npm. Windows release packaging also
requires PowerShell 7 (`pwsh`).

```bash
npm ci
npm run dev
npm run typecheck
npm run lint
npm test
npm run build:app
npm run build:unpacked
npm run build
npm run test:e2e:windows
```

`build:app` creates the electron-vite application bundles without an installer.
The `build:unpacked` and `build` commands target Windows x64 and require the pinned
portable runtime generated by `npm run runtime:build`. The runtime, NSIS, and Playwright
steps are Windows release gates and are not simulated on Linux.

## Architecture

- `electron/main/application`: explicit application lifecycle and composition.
- `electron/main/windows`: the only `BrowserWindow` registry and sender authority.
- `electron/main/ipc`: narrow handlers with sender, URL, payload, and result checks.
- `electron/main/infrastructure`: fixed production server config, encrypted secrets,
  redacted logging, and versioned settings.
- `electron/main/services`: bounded HTTP/OCR and card assets, Windows device identity,
  auth session, generation-fenced WebSocket/setup/monitor state machines, local-widget
  controller, source resolver, owned child-process supervisors, and one custom updater.
- `electron/main/domain`: internal-only capture selectors; renderer DTOs never contain
  Electron source IDs, HWNDs, process IDs, monitor indices, or executable paths.
- `python`: framed one-shot capture and trigger analysis workers plus a strict persistent
  JSON-lines monitor, pinned to `windows-capture==2.0.0`.
- `electron/preload`: independent main, auth, setup, and widget entry points.
- `shared/contracts`: Zod IPC contracts shared by main and preload.
- `shared/models` and `shared/errors`: boundary-safe models and errors.
- `renderer`: separate sandboxed auth, main, setup, and widget React entries with
  restrictive CSPs.

The renderer has no Node.js integration, filesystem access, tokens, full HWID,
generic IPC, or generic settings API. Auth and main windows expose separate narrow
preload APIs. Every call is runtime-validated on both sides, and main verifies the
sender WebContents ID and renderer URL through the one window registry.

Settings use strict Zod schemas, contain no secrets, and are written atomically to
`userData/settings.v1.json`. The pure defaults and migration helpers and the file
system boundary make the repository testable.

Local widget settings are strict per-user atomic files under
`userData/widget-settings.v1/`. Defaults enable auto-open and always-on-top, leave the
window movable, and use high opacity. Position and size are bounded for Windows and saved
after user move/resize events.

Capture configuration is separate and per server user under
`userData/capture-config.v1/`. It stores normalized regions, a compact trigger template,
non-sensitive source preference metadata, revision, and fingerprint. It never stores
screenshots, thumbnails, raw source IDs, HWNDs, tokens, or full executable paths. Setup
posts the exact schema-version 2 legacy projection to
`/api/user/me/ocr-regions` first, then atomically commits the local configuration.

The production origin is fixed to `https://api.artcsworld.xyz`; realtime uses
`wss://api.artcsworld.xyz/ws`. The refresh token is encrypted with Electron
`safeStorage` and written atomically to `userData/auth.v1.enc.json`. Access tokens
remain memory-only. If platform encryption is unavailable, authentication fails
closed and no plaintext fallback is written. Logout is local because the current
server has no revoke endpoint; `AuthSessionRevoker` is the explicit future seam.

Production HWID generation is Windows-only and uses one non-shell PowerShell/CIM
invocation. Non-production Linux development uses an explicitly constructed dev
provider and cannot silently instantiate the Windows production identity.

## Capture Setup

M3 implementation is complete. The Capture workspace explicitly enumerates Electron
window/display sources, uses opaque short-lived keys, and creates bounded lazy previews.
Starting setup captures the canonical frame before the setup window exists. The isolated
wizard maps pointer and numeric coordinates through a letterboxed contain transform and
stages Trigger, Normal, and Precise regions before analysis/review/commit.

Install the Python environment on Windows with Python 3.9 or newer:

```powershell
py -m venv .venv
.venv\Scripts\python -m pip install -r python\requirements.txt
$env:CR_TOOLS_PYTHON_PATH = "$PWD\.venv\Scripts\python.exe"
npm run dev
```

Production packaging includes a managed interpreter at
`resources/python-runtime/python.exe`. `scripts/Build-PortablePython.ps1` downloads the
pinned NuGet Python 3.11.9 package on Windows, installs only pinned wheels, runs `pip check`
and import smoke tests, removes caches/tests/bytecode where safe, and preserves discovered
license files. `scripts/New-RuntimeInventory.ps1` writes the bundled relative-path,
size, and SHA-256 inventory. The application never downloads Python at runtime. There is
no fake Linux runtime or alternate screenshot backend.

Display capture maps Electron displays to Windows device names only when a fixed,
per-monitor-aware PowerShell helper returns a unique exact physical-bounds match. Missing,
duplicate, or mismatched data remains unsupported rather than guessing monitor order.
Window capture uses a validated signed-64-bit decimal HWND with `window_hwnd`.

## Monitor And Results

M4 is implemented. Monitoring never auto-starts. Start performs auth, per-user preference,
capture-configuration, and exact durable-source preflight before spawning one owned Python
worker. Saved window preferences require an exact title and, when present, exact executable
label; zero and duplicate matches fail with actionable errors. Display resolution remains
fail-closed.

Monitor preferences are strict per-user atomic files under
`userData/monitor-preferences.v1/` with `fast`/`pol` defaults. The renderer can read or
change only search mode and deck mode while stopped. It receives a bounded `MonitorView`;
tokens, HWNDs, selectors, trigger templates, source images, raw OCR/model responses, and
unbounded server data never cross the preload boundary.

The private monitor protocol is JSON-lines protocol version 1. Main sends one validated
start command and an eventual stop command. Python emits monotonic, session-bound `ready`,
`action`, `fatal`, and `stopped` events. Action PNG base64 exists only on this private pipe,
is decoded immediately in main, and is never logged or sent to a renderer. Worker readiness
has an 8-second deadline; stop is graceful first and then uses Windows process-tree
termination. Stdout lines, stderr diagnostics, image dimensions/bytes, event ordering, and
session generations are bounded and validated.

The local trigger engine checks at 10 FPS with bitwise 64-bit aHash distance 18, normalized
128x128 grayscale, ORB BF/Hamming cross-check at distance 55 and 10 matches or NCC at 0.72,
soft `+1/-0.5` confirmations to 2, and a 15-second cooldown. It crops the configured fast or
precise source-local physical-pixel region and emits a bounded PNG. OCR uses only
`POST https://api.artcsworld.xyz/api/ocr/process`, a 150-second deadline, owner
cancellation, and one auth refresh after 401. Player found, player not found, recognition
failure, service/network failure, and blocked auth remain distinct.

Development source execution uses `python/monitor_engine.py`. Production packages include
the managed Python runtime and verify its bounded size/SHA-256 inventory before the first
Python spawn. Linux does not simulate a successful capture; only pure protocol, algorithm,
process, OCR, and supervisor tests are cross-platform.

## Local Widget

M5 is implemented. `WidgetController` owns one generation-safe monitor result subscription
and opens the dedicated widget only for a new `player_found` result when auto-open is
enabled. Manually opening the widget clearly shows empty, not-found, recognition-failure,
or service-error state. Closing the widget never quits the application; logout and auth
loss close it.

`WindowCoordinator` remains the only `BrowserWindow` registry. The widget has a dedicated
sandboxed preload and renderer entry, production devtools disabled, blocked navigation and
popups, and a CSP with `connect-src 'none'`. Its preload exposes only view polling, indexed
card asset lookup, allowed settings updates, and hide. The view strips card URLs and never
contains tokens, capture selectors, raw OCR responses, or screenshots.

Card images are optional. `ImageAssetService` resolves result/deck/card indices against a
currently retained monitor result and only accepts exact HTTPS host
`api-assets.clashroyale.com`. It performs manual redirect validation, a five-second
timeout, 512 KiB response limit, PNG/JPEG/WebP signature and dimension checks, sends no
credentials or referrer, and stores only a bounded in-memory LRU. Any failure displays card
initials and name. Renderers cannot provide an asset URL or perform direct HTTP requests.

Widget controls are keyboard-focusable and labeled: pin, position lock, compact mode,
opacity, deck tabs, and hide. The Settings page exposes the same behavior and honestly
also owns manual update check, download, cancellation, and install controls.

## Streamer Workspace

M6 is implemented for the active V1 parity scope. The tabbed Streamer workspace supports
Twitch connect/status/disconnect, automatic predictions, result-region calibration, automatic
stream titles and Clash accounts, deck sharing, and adaptive OBS opponent/stats overlays.
Alerts, Discord, the old generic OBS placeholder, and `smart_predictions` are intentionally not
ported.

All streamer management requests originate in main through an authenticated API seam that keeps
the app JWT memory-only and retries once after a 401 refresh. The renderer receives one strict,
bounded `StreamerView`; Twitch credentials, widget tokens, complete OBS URLs, screenshots, raw
server payloads, and model output do not cross preload. OBS URLs can only be copied by a typed
`stats`/`opponent` command. Mock previews are local React fixtures and never call the public
preview endpoint.

OAuth opens only a server-returned HTTPS URL whose exact host is `id.twitch.tv`, using Electron's
system-browser `shell.openExternal`. The callback remains server-owned, and main performs bounded,
non-overlapping status polling while the Streamer page is active or a connection attempt is
pending. No remote `BrowserWindow` or `window.open` is used.

Prediction preferences and result profiles are strict per-user atomic files under
`userData/prediction-preferences.v1/` and `userData/prediction-result-config.v1/`. Result setup
re-resolves the current main capture target, captures before opening the setup window, analyzes
the result trigger locally, then saves trigger and data endpoints in order. Because the server has
no transaction across those two endpoints, a second-write failure is reported as partial remote
state and the local profile remains inactive.

Predictions reuse the single monitor process. Normal battle triggers notify the server through a
serialized main-only coordinator. The optional result profile has one confirmation and a 60-second
cooldown; its cropped PNG is emitted on the private process pipe and posted as multipart with a
150-second deadline, never exposed to a renderer. Start rolls the server back if local monitor
configuration fails. Stop removes prediction runtime while preserving a monitor that was already
running.

Known server limitations are not hidden by client fallbacks. The OAuth state now uses one-time,
expiring server records, but the previously exposed Twitch client secret must still be rotated.
V2 also keeps the system-browser flow outside Electron. OBS URLs remain bearer capabilities under
the server's widget-token model; rotating the token invalidates existing browser sources.

## Updates And Releases

M7 implements exactly one custom updater; `electron-updater` is not installed. It runs only
in packaged Windows builds. Development and non-Windows builds report that updating is
unsupported instead of simulating success. Startup performs a delayed check only and never
downloads or installs automatically.

The fixed manifest is
`https://updates.artcsworld.xyz/downloads/v2/manifest.json`. Main accepts only strict schema
version 1, verifies its canonical payload with Ed25519, and then permits only an exact
`https://updates.artcsworld.xyz/downloads/v2/CR_Tools_V2_Setup_<x.y.z>.exe` artifact.
Redirects remain on that origin/path, downloads are bounded and atomic, and size plus SHA-512
are verified before READY and again immediately before launch. The renderer never receives a
URL, signature, hash, or filesystem path.

The checked production public key is `resources/update-public-key.pem`. Its SHA-256 SPKI
fingerprint is `2a16488a2a16440e6c1ac19f82f9b262b7e9154d0851e3dbbac0be8d9b612d99`.
Manifest signing is not Authenticode. By product decision, releases have no Windows
Authenticode certificate, so SmartScreen can show an unavoidable "Unknown publisher" warning.
Neither the Ed25519 manifest nor electron-builder executable resource editing would remove
that warning.

Run `./publish-update.sh` for the interactive one-command Windows build flow. Test mode downloads a
verified installer artifact without deployment; release mode signs and publishes only after an
explicit confirmation. See `docs/RELEASE_RUNBOOK.md` for setup, key rotation, artifact verification,
deployment, and rollback procedures.
